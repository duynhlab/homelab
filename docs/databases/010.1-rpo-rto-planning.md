# PostgreSQL RPO/RTO Planning

Child playbook of the [PostgreSQL Disaster Recovery Plan](./010-drp.md). The DRP
is the system of record; this page is the **planning view**: it maps each data
tier to a concrete cluster, states the **target** RPO/RTO, and contrasts it with
what the platform actually delivers **today**. It deliberately does not re-derive
the math — see [006-backup-strategy.md](./006-backup-strategy.md) for the RPO/RTO
formulas and [010-drp.md](./010-drp.md#rporto-matrix) for the scenario matrix.

- **RPO** (Recovery Point Objective) — how much data you can afford to lose,
  measured backwards from the incident. Bounded by replication and WAL-archive lag.
- **RTO** (Recovery Time Objective) — how long recovery may take, measured forward
  from the incident to validated service. Proven only by drills
  ([010.2](./010.2-restore-and-failover-drills.md)).

## Data tiers → clusters

| Tier | Meaning | Cluster(s) | Operator |
|------|---------|-----------|----------|
| **T0 — critical transactional** | Money/checkout path; loss is unacceptable | `product-db` (`product`, `cart`, `order`, `payment`) | CloudNativePG |
| **T1 — important user-facing** | Auth/identity; brief loss tolerable if documented | `auth-db` | CloudNativePG |
| **T2 — supporting** | Notification/shipping/review metadata | `shared-db` | CloudNativePG |
| **T3 — orchestration state** | Temporal workflow history | `temporal-db` | CloudNativePG |

## Target RPO/RTO by tier

These are the objectives the design is built to meet once drills are recorded.

| Tier | Target RPO | Target RTO | Primary recovery path |
|------|-----------|-----------|-----------------------|
| T0 `product-db` | **0** for HA failover; **≤ 5 min** for DR promotion | **< 1 min** failover; **minutes** DR promotion | HA failover → DR promotion → PITR |
| T1 `auth-db` | **0** for HA failover; **≤ 5 min** for restore | **< 1 min** CNPG failover; **manual** for restore | CNPG HA failover → Barman restore |
| T2 `shared-db` | **≤ 1 backup interval** | **Manual restore time** (owner-accepted) | Barman restore-to-new-cluster |
| T3 `temporal-db` | **Undefined** (no backup — see gap) | Recreate + workflow replay | Rebuild cluster |

## As-built RPO/RTO today

What the current homelab manifests actually guarantee. The gap between this and
the target table above is the DR backlog.

| Cluster | RPO driver (as-built) | Effective RPO | RTO posture | Evidence |
|---------|----------------------|---------------|-------------|----------|
| `product-db` | 3 instances, sync quorum `ANY 1`; `archive_timeout 5m`; daily + every-6h base backups → `s3://pg-backups-cnpg/product-db/` (30-day retention) | **0** on quorum-ack commits; **≤ 5 min** to DR replica | HA failover is automatic; DR promotion + PITR are **not yet drill-recorded** | ⏳ drills pending |
| `product-db-replica` | Follows `product-db` via object-store recovery; own backups (7-day retention) | Tracks primary minus replay lag | Promote via `replica.enabled: false` | ⏳ promotion drill pending |
| `auth-db` | 3 instances, sync quorum `ANY 1`; Barman → `s3://pg-backups-cnpg/auth-db/` (30-day retention) | **0** on quorum-ack commits | CNPG auto-failover; restore is manual | ⏳ restore drill pending |
| `shared-db` | **1 node (SPOF)**; Barman → `s3://pg-backups-cnpg/shared-db/` (30-day retention) | Since last WAL archive | No DB failover; manual restore only | ⏳ |
| `temporal-db` | **No backups, no WAL archiving** | **n/a — total loss on PVC failure** | Recreate cluster; Temporal replays history from its own DB (which is what's lost) | ❌ unprotected |

## How targets map to backup cadence

RPO and RTO are not free-floating numbers — they are consequences of concrete knobs:

- **RPO is set by the archive/replication lag.** For `product-db`, synchronous
  quorum makes acknowledged commits RPO-0 for in-cluster failover. For anything
  recovered from the object store, RPO is bounded by `archive_timeout` (**5 min**)
  plus upload time — a WAL segment not yet in RustFS is not recoverable.
- **Base-backup frequency sets the PITR floor + the restore baseline.** Daily +
  every-6h base backups keep the WAL-replay distance (and therefore restore time)
  short. Longer gaps between base backups mean more WAL to replay at restore → higher RTO.
- **RTO is the sum of detect + decide + restore + replay + validate + cut-over.**
  See the RTO breakdown in [006-backup-strategy.md](./006-backup-strategy.md). The
  only component you cannot estimate from config is *validate + cut-over* — that's
  what the drills in [010.2](./010.2-restore-and-failover-drills.md) measure.

## Known gaps

These widen the gap between target and as-built; tracked in
[010-drp.md → Known Gaps](./010-drp.md#known-gaps-and-next-improvements).

- **`temporal-db` is unprotected** — no `ScheduledBackup`, no WAL archiving. Its
  RPO/RTO are undefined and a PVC loss is unrecoverable. Follow-up: add a Barman
  `ObjectStore` + `ScheduledBackup` (mirroring `product-db`) so T3 gets a real target.
- **`shared-db` is a single-node SPOF** — no DB-level failover; scaling
  `spec.instances` to 3 would give it CNPG HA (the historical Zalando scaling
  runbook [zalando-ha-scaling.md](./runbooks/zalando-ha-scaling.md) is kept for
  reference only).
- **No recorded drills** — every "Estimated/pending" above stays an estimate until
  [010.2](./010.2-restore-and-failover-drills.md) produces signed evidence.
