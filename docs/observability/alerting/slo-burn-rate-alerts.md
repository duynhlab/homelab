## 1. The method: multi-window, multi-burn-rate

Two failure modes need different sensitivity:

| Mode | Signature | Right detector |
|---|---|---|
| **Sudden** outage | 5xx jumps from 0 â†’ 90% in seconds | Short window (5 min), high burn rate |
| **Slow** degradation | 5xx drifts up over hours | Long window (1â€“6 h), moderate burn rate |

A single threshold on a single window can't satisfy both â€” pick a low threshold and you flap; pick a high one and you miss the slow burn. The **multi-window multi-burn-rate** pattern from [Google SRE Workbook ch. 5](https://sre.google/workbook/alerting-on-slos/) AND-gates a short window with a long window so each alert needs *both* time scales to agree before firing.

### Burn rate, in one line

```
burn_rate = (actual error fraction during window) / (1 - SLO objective)
```

`burn_rate = 1` means "you'll burn the whole 30-day budget exactly at day 30". `burn_rate = 14.4` means "you'll burn it in `30/14.4 â‰ˆ 2 days`".

### Why exactly 14.4Ã— and 6Ã—?

Numbers come from the workbook and are calibrated for a **30-day** SLO window:

| Severity | Long window | Burn rate | Budget consumed if it persists for the full long window | Time-to-exhaustion if rate stays |
|---|---|---|---|---|
| **Page** | 1 h | **14.4Ã—** | 2% of 30-day budget in 1 h | ~2 days |
| **Ticket** | 6 h | **6Ã—** | 5% of 30-day budget in 6 h | ~5 days |

The short window (5 min for page, 30 min for ticket) just confirms the burn is happening *right now* â€” both sides must be over the threshold.

For a deeper derivation see the [SRE Workbook chapter on alerting on SLOs](https://sre.google/workbook/alerting-on-slos/) â€” Sloth implements the "Alert 6 (Multiwindow, Multi-Burn-Rate Alerts)" variant.

## 2. What Sloth generates per SLO

Each `PrometheusServiceLevel` produces a `PrometheusRule` containing **recording rules** + **2 alerts per SLO**. With our 8 services Ã— 3 SLOs (availability, latency, error-rate) that's **24 SLOs / 48 alerts**.

### Recording rules (per SLO, per window)

```promql
# Sliding-window SLI error ratio over 5m / 30m / 1h / 2h / 6h / 1d / 3d
slo:sli_error:ratio_rate5m{sloth_service="auth", sloth_slo="availability"}
slo:sli_error:ratio_rate1h{sloth_service="auth", sloth_slo="availability"}
slo:sli_error:ratio_rate6h{sloth_service="auth", sloth_slo="availability"}

# Period-aggregated metrics
slo:current_burn_rate:ratio{sloth_service="auth", sloth_slo="availability"}
slo:error_budget_remaining:ratio{sloth_service="auth", sloth_slo="availability"}
slo:objective:ratio{sloth_service="auth", sloth_slo="availability"}
```

### The 2 alerts (Sloth default labels)

For each SLO Sloth emits exactly two alert rules:

| Alert name pattern | Severity label | Short window | Long window | Burn rate | Wakes you up? |
|---|---|---|---|---|---|
| `<service>HighErrorRate` (page) | `severity: page` | 5 min | 1 h | 14.4Ã— | **Yes** |
| `<service>HighErrorRate` (ticket) | `severity: ticket` | 30 min | 6 h | 6Ã— | No â€” business-hours follow-up |

The alert name is reused for the SLO category (the SLI determines the meaning):

| SLO | Page-alert name (default) | What "high error rate" means here |
|---|---|---|
| Availability | `<svc>HighErrorRate` | 5xx fraction is burning the availability budget |
| Latency | `<svc>HighLatency` | Fraction of requests slower than 500 ms is burning the latency budget |
| Error rate | `<svc>HighOverallErrorRate` | 4xx + 5xx fraction is burning the error-rate budget |

(Names are configurable in the `mop` chart values; defaults above match the chart at the time of writing.)

### Standard alert labels and annotations

Every Sloth-generated alert carries:

| Field | Source | Used for |
|---|---|---|
| `sloth_service` | `PrometheusServiceLevel.spec.service` | Routing & dashboards |
| `sloth_slo` | SLO name (`availability` / `latency` / `error-rate`) | Filter in Karma / Sloth UI |
| `sloth_severity` | `page` or `ticket` | Routing in VMAlertmanager |
| `severity` | mirrors `sloth_severity` | VMAlertmanager standard label |
| `team` | `sloth.extraLabels.team` (we set `platform`) | Ownership |
| `summary` / `description` | rendered with the burn rate and budget remaining | Alert payload |

## 3. Configuration â€” per-service overrides

SLOs are wired through the [`mop` Helm chart](https://github.com/duyhenryer/charts/tree/main/charts/mop) â€” see [`getting_started.md`](../slo/getting_started.md) for the full flow. Alert behaviour is controlled by these values:

```yaml
slo:
  enabled: true
  team: platform              # propagates as `team` label
  env: monitoring             # propagates as `env` label
  availability:
    objective: 99.5           # changes the 14.4Ã—/6Ã— anchors automatically
  latency:
    objective: 95.0
    threshold: "0.5"          # 500 ms â€” the bucket le="..."
  errorRate:
    objective: 99.0
```

**Tightening the SLO** (e.g. `99.5 â†’ 99.9`) **automatically tightens the alerts** â€” the burn-rate maths is anchored on `1 - objective`, so 14.4Ã— of a smaller error budget triggers on a smaller absolute error rate. You don't (and shouldn't) hand-edit the burn-rate thresholds.

To change *labels* on the generated alerts (route to a different Slack channel, mark as critical), edit the chart values; do not edit Sloth's output (it gets regenerated).

## 5. On-call handling

When a Sloth-generated alert fires, work the problem in this order:

### Page alert (`severity=page`) â€” wake-up

1. **Acknowledge in Karma** within 5 minutes. Stops re-pages.
2. **Identify scope**:
   - Open Sloth UI â†’ service detail. Is *one* SLO burning, or all three? Is *one* service burning, or many?
   - One service, one SLO â†’ a code/dependency issue specific to that path.
   - All services, availability SLO â†’ infra-wide (gateway, network, DB, control plane).
3. **Correlate with deploys / config changes**:
   - Flux UI ([ui.duynh.me](http://ui.duynh.me)) â€” recent reconciliations.
   - Service repo Git log around the burn start time.
4. **Drill into Layer-1 alerts** for the same service in Karma â€” `MicroserviceHighErrorRate`, `MicroserviceHighP95Latency` etc. usually point at the proximate cause faster than the SLO alert.
5. **Mitigate**, then **silence** the SLO alert in Karma with a clear comment + expiry. Don't leave silences open-ended.

### Ticket alert (`severity=ticket`) â€” next business hours

1. Confirm the burn is real (Sloth UI â†’ SLI chart). Transient bumps that didn't hit page severity can resolve on their own.
2. If the trend continues, file a ticket and treat it as planned reliability work.
3. Check [`error_budget_policy.md`](../slo/error_budget_policy.md) â€” if the budget is below 20%, follow the policy gates on deploys.

### Per-SLI starter checks

| SLO | First Grafana panel | First PromQL query |
|---|---|---|
| Availability | "Microservice errors" â†’ 5xx by endpoint | `sum by(handler)(rate(request_duration_seconds_count{app="<svc>",code=~"5.."}[5m]))` |
| Latency | "Microservice latency" â†’ p95 by endpoint | `histogram_quantile(0.95, sum by(le,handler)(rate(request_duration_seconds_bucket{app="<svc>"}[5m])))` |
| Error rate | "Microservice errors" â†’ 4xx & 5xx by endpoint | same as availability with `code=~"4..|5.."` |

## 6. Testing & verification

### Confirm the rules exist after you enable an SLO

```bash
# PrometheusServiceLevel reconciled OK
kubectl get prometheusservicelevel -A
kubectl describe prometheusservicelevel <svc> -n monitoring   # GEN OK=true

# Sloth generated the PrometheusRule
kubectl get prometheusrule -n monitoring -l app.kubernetes.io/managed-by=sloth

# VM Operator converted it to a VMRule
kubectl get vmrule -n monitoring -l app.kubernetes.io/managed-by=sloth
```

### Confirm VMAlert picked it up

Open [vmalert.duynh.me](http://vmalert.duynh.me) and search for the alert name (`<svc>HighErrorRate` etc.) â€” you should see the rule, the last evaluation timestamp, and the current value.

### Force the alert to fire (synthetic load)

Real traffic is the safest source. If you must synthesise, use the existing k6 suite documented in [`docs/testing/k6.md`](../../testing/k6.md) â€” it already targets the public endpoints via Kong. **Do not** invent endpoints (`/api/error`, `/api/slow`) â€” none exist in the services. To exercise the availability SLO, point k6 at a real handler and induce errors via the chaos test scenarios shipped with the suite.

### Test the routing without real traffic

```bash
# Send a fake alert directly to VMAlertmanager
kubectl -n monitoring port-forward svc/vmalertmanager-victoria-metrics 9093 &

curl -X POST http://localhost:9093/api/v2/alerts -H 'Content-Type: application/json' -d '[{
  "labels": { "alertname": "AuthHighErrorRate", "severity": "page", "sloth_service": "auth", "sloth_slo": "availability" },
  "annotations": { "summary": "synthetic test" },
  "startsAt": "2026-01-01T00:00:00Z",
  "endsAt":   "2026-01-01T00:05:00Z"
}]'

# Verify in Karma
open http://karma.duynh.me
```

## 7. Known limitations / gotchas

- **Burn-rate maths assumes a stable traffic volume.** A service with very low RPS will see one bad request blow the short-window burn rate. Sloth mitigates this by computing the SLI as a ratio over a full window, but if you have <1 RPS sustained, expect noise.
- **Mixed-traffic services** (background jobs piggybacking on the same `request_duration_seconds`) pollute the SLI. Filter at the SLI level (`handler!~"â€¦"`) if needed.
- **The SLI is a 30-day rolling window.** A burn from 25 days ago still counts toward today's budget. Don't be surprised if the budget is low even when current traffic is healthy.
- **Routing is unfinished** â€” VMAlertmanager has only a `default` receiver. Until Slack/PagerDuty receivers are wired, "page" severity = "appears in Karma faster". Track via the [Future Roadmap in `alerting/README.md`](./README.md#future-roadmap).
- **Dashboards in VMAlert UI** show rule definitions only; "for live SLI shape, prefer the Sloth UI or Grafana SLO dashboards."
