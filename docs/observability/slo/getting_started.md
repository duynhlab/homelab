# Getting Started with SLOs

## Overview

SLOs are automatically generated by the `mop` Helm chart when `slo.enabled: true` is set in a service's HelmRelease values. The Sloth Operator then generates PrometheusRules (recording + alerting rules) following Google SRE multi-window multi-burn-rate pattern.

## Prerequisites

1. **Sloth Operator** deployed (part of `controllers-local` Kustomization)
2. **Prometheus** running with ServiceMonitor support
3. **Service** deployed via the `mop` Helm chart

## Enable SLO for a Service

### Step 1: Add `slo.enabled` to Service Values

In the domain ResourceSet templates (`kubernetes/apps/domains/*-rs.yaml`), SLOs are already enabled for all backend services via `slo.enabled: true`. To override defaults for a specific service, customize the service's InputProvider (`kubernetes/apps/services/<name>.yaml`) or the domain template. For standalone HelmReleases:

```yaml
spec:
  values:
    name: my-service

    slo:
      enabled: true
```

This uses the default SLO targets (99.5% availability, 95% latency, 99% error rate). To override:

```yaml
    slo:
      enabled: true
      availability:
        objective: 99.9    # stricter availability
      latency:
        objective: 99.0    # stricter latency
        threshold: "0.3"   # 300ms instead of 500ms
      errorRate:
        objective: 99.5    # stricter error rate
```

### Step 2: Deploy

```bash
make flux-push && make flux-sync
```

Flux reconciles the HelmRelease, Helm renders the `PrometheusServiceLevel` CRD, and Sloth generates the PrometheusRules automatically.

### Step 3: Verify

```bash
# Check PrometheusServiceLevel was created
kubectl get prometheusservicelevel -n monitoring

# Check Sloth generated PrometheusRules
kubectl get prometheusrule -n monitoring | grep sloth

# Describe to see GEN status
kubectl describe prometheusservicelevel my-service -n monitoring
# Look for: GEN OK=true

# Check SLO metrics in Prometheus
kubectl port-forward -n monitoring svc/kube-prometheus-stack-prometheus 9090:9090
# Query: slo:sli_error:ratio_rate5m{sloth_service="my-service"}
```

## Default SLO Values

Defined in the mop chart's `values.yaml` ([`duyhenryer/charts` repo](https://github.com/duyhenryer/charts/blob/main/charts/mop/values.yaml)):

```yaml
slo:
  enabled: false          # opt-in per service
  team: "platform"
  env: "monitoring"
  availability:
    objective: 99.5       # 99.5% non-5xx
  latency:
    objective: 95.0       # 95% < 500ms
    threshold: "0.5"      # 500ms threshold
  errorRate:
    objective: 99.0       # 99% non-4xx/5xx
```

## How ServiceMonitor Discovery Works

The central ServiceMonitor (`kubernetes/infra/configs/monitoring/servicemonitors/microservices.yaml`) automatically discovers services:

```yaml
namespaceSelector:
  any: true
selector:
  matchLabels:
    component: api    # All mop chart services have this label
```

No manual namespace registration is needed. Any service deployed via the `mop` chart gets the `component: api` label and is automatically scraped.

## Verification Checklist

After enabling SLO for a new service:

- [ ] `kubectl get prometheusservicelevel -n monitoring` shows the service
- [ ] `kubectl describe prometheusservicelevel <service> -n monitoring` shows `GEN OK=true`
- [ ] `kubectl get prometheusrule -n monitoring | grep sloth` shows generated rules
- [ ] Prometheus query `slo:sli_error:ratio_rate5m{sloth_service="<service>"}` returns data
- [ ] Grafana SLO dashboards show the new service

## Troubleshooting

### PrometheusServiceLevel not created

Check that the HelmRelease was reconciled:
```bash
flux get helmrelease -n <namespace>
```

### GEN OK=false

Check Sloth Operator logs:
```bash
kubectl logs -n monitoring -l app.kubernetes.io/name=sloth
```

### No SLO metrics in Prometheus

Verify the service is being scraped:
```bash
# Check ServiceMonitor targets in Prometheus
kubectl port-forward -n monitoring svc/kube-prometheus-stack-prometheus 9090:9090
# Open: http://localhost:9090/targets -- look for microservices job
```

Verify the service emits `request_duration_seconds`:
```promql
request_duration_seconds_count{app="<service>"}
```
