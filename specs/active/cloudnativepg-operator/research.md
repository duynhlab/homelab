## Executive Summary

CloudNativePG (CNPG) is a Kubernetes operator for PostgreSQL that provides declarative management of PostgreSQL clusters. Unlike Zalando Postgres Operator, **CloudNativePG does NOT have built-in cross-namespace secret management**. Secrets must be manually created before cluster bootstrap and manually copied to target namespaces if multiple services in different namespaces need access.

**Key Finding**: CloudNativePG requires **pre-created secrets** for bootstrap (unlike Zalando which auto-generates secrets). For multi-namespace access (e.g., transaction-db used by both `cart` and `order` services), secrets must be manually copied or managed via external tools.

**Current Issue**: `transaction-db-secret` is created in `cart` namespace, but `order` service (in `order` namespace) cannot access it due to Kubernetes namespace isolation.

**Immediate Solution (APPROVED)**: Create separate YAML files for secrets in each namespace (`transaction-db-secret-cart.yaml`, `transaction-db-secret-order.yaml`). Update deployment script to apply YAML files. This approach is acceptable for personal/learning projects where secrets in git are acceptable.

**Long-Term Solution**: **External Secrets Operator** with PushSecret feature for production deployments. Provides centralized secret management, automatic password rotation, and multi-namespace synchronization.

**Production-Ready Focus**: This research covers comprehensive production-ready patterns including:
- Secret management (YAML files, External Secrets Operator, Managed Roles)
- High Availability configuration (3+ instances, synchronous replication)
- Monitoring integration (PodMonitor, Prometheus, Grafana)
- Advanced CloudNativePG features (declarative extensions, logical replication, replica clusters, hibernation, fencing)
- Backup strategies (WAL archiving, volume snapshots, PITR)
- Performance tuning and security best practices

**CloudNativePG Secret Creation Pattern:**
```bash
# Transaction database secret (cart + order)
if ! kubectl get secret transaction-db-secret -n cart &> /dev/null; then
```

**Key Differences from Zalando:**

### Zalando vs CloudNativePG Secret Management

| Feature | Zalando Operator | CloudNativePG |
|---------|------------------|---------------|

## External Solutions

### Option 1: Manual Secret Copy (Current Pattern)

**What it is:** Copy secret from source namespace to target namespace using `kubectl`.

**How it works:**
1. Create secret in source namespace (`cart`)
2. Copy secret to target namespace (`order`) using `kubectl get secret ... -o yaml | sed ... | kubectl apply`
3. Update script to automate copy step

**Implementation:**
```bash
# In scripts/04-deploy-databases.sh, after creating transaction-db-secret in cart namespace:
# Copy secret to order namespace
if ! kubectl get secret transaction-db-secret -n order &> /dev/null; then
    kubectl get secret transaction-db-secret -n cart -o yaml | \
        sed 's/namespace: cart/namespace: order/' | \
        sed '/resourceVersion:/d' | sed '/uid:/d' | sed '/creationTimestamp:/d' | \
        kubectl apply -f -
    echo "Copied transaction-db-secret to order namespace"
fi
```

**Pros:**
- ✅ Simple and straightforward
- ✅ No additional tools required
- ✅ Works immediately
- ✅ Low complexity

**Cons:**
- ⚠️ Manual sync required when password changes
- ⚠️ Two secrets to maintain (potential drift)
- ⚠️ Not automatic (requires script execution)
- ⚠️ No automatic updates when source secret changes

**Implementation Complexity:** Low
**Team familiarity:** High (standard kubectl operations)
**Fit for our use case:** High - Quick fix for current issue

### Option 2: External Secrets Operator (Production-Ready)

**What it is:** Kubernetes operator that syncs secrets from external secret stores (Vault, AWS Secrets Manager, etc.) to Kubernetes secrets.

**How it works:**
1. Install External Secrets Operator
2. Create `SecretStore` resource pointing to Vault/AWS Secrets Manager
3. Create `ExternalSecret` CRD in each namespace that needs the secret
4. Operator automatically syncs secrets from external store to Kubernetes secrets
5. When external secret changes, Kubernetes secret is automatically updated

**Architecture:**
```
Vault/AWS Secrets Manager
    ↓ (password rotation)
External Secrets Operator
    ↓ (syncs to K8s secrets)
Kubernetes Secret (cart namespace)
Kubernetes Secret (order namespace)
    ↓ (consumed by)
Cart Service Pod
Order Service Pod
```

**Configuration Example:**
```yaml
# SecretStore (cluster-wide or namespace-scoped)
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: vault-backend
  namespace: cart
spec:
  provider:
    vault:
      server: "https://vault.example.com"
      path: "secret"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "external-secrets-operator"
---
# ExternalSecret in cart namespace
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: transaction-db-secret
  namespace: cart
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: SecretStore
  target:
    name: transaction-db-secret
    creationPolicy: Owner
  data:
  - secretKey: username
    remoteRef:
      key: postgres/transaction-db
      property: username
  - secretKey: password
    remoteRef:
      key: postgres/transaction-db
      property: password
---
# ExternalSecret in order namespace (same secret, different namespace)
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: transaction-db-secret
  namespace: order
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: SecretStore
  target:
    name: transaction-db-secret
    creationPolicy: Owner
  data:
  - secretKey: username
    remoteRef:
      key: postgres/transaction-db
      property: username
  - secretKey: password
    remoteRef:
      key: postgres/transaction-db
      property: password
```

**Pros:**
- ✅ **Centralized secret management** - All secrets in Vault/AWS Secrets Manager
- ✅ **Automatic password rotation** - Change in Vault automatically syncs to K8s
- ✅ **Audit trail** - Vault logs all secret access
- ✅ **Multi-namespace support** - Same secret synced to multiple namespaces
- ✅ **Production-ready** - Industry standard approach
- ✅ **Security** - Secrets encrypted at rest in Vault
- ✅ **Compliance** - Meets security policy requirements

**Cons:**
- ⚠️ Requires Vault/AWS Secrets Manager setup
- ⚠️ Additional operator to install and maintain
- ⚠️ More complex initial setup
- ⚠️ Learning curve for team

**Implementation Complexity:** Medium-High
**Team familiarity:** Low (new tool, needs training)
**Fit for our use case:** High - Best for production, but requires infrastructure setup

**Configuration Example:**

**Pros:**
- ✅ Automatic replication

**Cons:**

### Option 5: CloudNativePG Managed Roles (Future Consideration)

**What it is:** CloudNativePG's `managed.roles` feature allows declarative PostgreSQL role/user management, but secrets are still created manually.

**How it works:**
1. Define roles in Cluster CRD `spec.managed.roles`
2. Operator creates PostgreSQL users automatically
3. **BUT**: Secrets must still be created manually in Kubernetes
4. Operator does NOT create Kubernetes secrets for managed roles

**Configuration Example:**
```yaml
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: transaction-db
spec:
  managed:
    roles:
    - name: cart
      login: true
      passwordSecret:
        name: transaction-db-secret
        key: password  # Secret must exist BEFORE cluster creation
      comment: "Database user for cart and order services"
    - name: monitoring
      login: true
      passwordSecret:
        name: monitoring-secret
        key: password
      inRoles:
        - pg_monitor  # Grant monitoring role
```

**Key Features:**
- **Declarative Role Management**: Define users in Cluster CRD
- **Automatic User Creation**: Operator creates PostgreSQL users
- **Role Inheritance**: Use `inRoles` to grant PostgreSQL roles (e.g., `pg_monitor`)
- **Password Management**: References existing Kubernetes secrets
- **Comment Support**: Add descriptions to roles

**Key Limitation:** `passwordSecret` references an existing Kubernetes secret - operator does NOT create it. Still requires manual secret creation.

**Use Cases:**
- **Monitoring Users**: Create `pg_monitor` role for Prometheus exporters
- **Application Users**: Declarative user management
- **Role-Based Access**: Grant specific PostgreSQL roles to users

**Conclusion:** Managed roles help with PostgreSQL user management but do NOT solve the cross-namespace secret problem. Useful for declarative user management and role assignment.

## Advanced CloudNativePG Features (Production-Ready)

### 1. Declarative Extensions Management

**What it is:** CloudNativePG 1.28+ supports loading PostgreSQL extensions dynamically from OCI-compliant container images without rebuilding base images.

**How it works:**
- Extensions are mounted as read-only volumes into PostgreSQL pods
- No need to rebuild PostgreSQL images when adding extensions
- Supports popular extensions: pgvector, PostGIS, pg_stat_statements, etc.

**Configuration Example:**
```yaml
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: transaction-db
spec:
  postgresql:
    extensions:
    - name: pgvector
      version: "0.5.0"
      repository: ghcr.io/cloudnative-pg/postgresql-extensions
```

**Use Cases:**
- **pgvector**: Vector similarity search for AI/ML applications
- **PostGIS**: Geographic data support
- **pg_stat_statements**: Query performance monitoring
- **Custom extensions**: Load any PostgreSQL extension dynamically

**Production Benefits:**
- ✅ Faster deployment (no image rebuild)
- ✅ Flexible extension management
- ✅ Version control for extensions
- ✅ Easy extension updates

### 2. Logical Replication Slot Synchronization ⚠️ RELATED TO HA

**What it is:** CloudNativePG automatically synchronizes logical decoding (replication) slots across all nodes in HA cluster, ensuring seamless continuation after failover.

**⚠️ IMPORTANT: This feature is DIRECTLY RELATED to HA configuration.**

**Why it matters for HA:**
- **Without synchronization**: Logical replication slots are lost during failover, causing data loss for CDC clients (Debezium, Kafka Connect)
- **With synchronization**: Slots are automatically synchronized across all replicas, ensuring zero data loss during failover
- **HA Requirement**: This feature is essential for production HA deployments that use logical replication or CDC

**How it works:**
1. Enable logical decoding slot synchronization in cluster config
2. Operator manages slot states during failover/switchover
3. Prevents slot invalidation and data loss for logical replication clients
4. Slots are automatically replicated to all standby nodes

**Configuration Example (for HA 3-node cluster):**
```yaml
apiVersion: postgresql.cnpg.io/v1
kind: Cluster

  name: transaction-db
  namespace: cart
spec:
  instances: 3  # HA configuration
  replicationSlots:
    highAvailability:
      synchronizeLogicalDecoding: true  # ⚠️ REQUIRED for HA with CDC
  postgresql:
    parameters:
      wal_level: 'logical'  # Required for logical replication
      hot_standby_feedback: 'on'  # Helps with replication lag
      sync_replication_slots: 'on'  # PostgreSQL 17+ (native sync)
      # For PostgreSQL 15/16, CloudNativePG handles sync automatically
```

```yaml
  - name: vault-backend

  - match:
      secretKey: password
      remoteRef:
        key: postgres/transaction-db
        property: password
```

**Integration with CloudNativePG:**

```yaml
      method: any  # Quorum-based

    size: 100Gi
```

### PodMonitor Configuration Options

**⚠️ IMPORTANT UPDATE (2026-01-02):** The `spec.monitoring.enablePodMonitor` field is **deprecated** and will be removed in a future version of CloudNativePG Operator. The recommended approach is to **manually create and manage PodMonitor resources** for each cluster.

**Reference:** [CloudNativePG Monitoring Documentation](https://cloudnative-pg.io/docs/1.28/monitoring)

#### Deprecation Notice: Built-in PodMonitor

**Status:** ⚠️ **DEPRECATED** - Will be removed in future CloudNativePG versions

The `.spec.monitoring.enablePodMonitor` field in the Cluster resource is now deprecated. CloudNativePG strongly recommends migrating to manual PodMonitor management.

**Why the change?**
- Ensures complete ownership of monitoring configuration
- Prevents operator from managing or overwriting PodMonitor resources
- Provides full control over monitoring lifecycle
- Aligns with Kubernetes best practices (explicit resource management)

**Migration Path:**
1. Set `.spec.monitoring.enablePodMonitor` to `false` (or remove the field)
2. Manually create PodMonitor resources for each cluster
3. Verify Prometheus Operator discovers the manual PodMonitors

#### Recommended Approach: Manual PodMonitor (Production-Ready)

**Status:** ✅ **RECOMMENDED** - Official CloudNativePG recommendation

The recommended approach is to **manually create and manage a PodMonitor** for each CloudNativePG cluster. This method provides full control over monitoring configuration and lifecycle.

**Implementation:**
- Files: `k8s/prometheus/podmonitors/podmonitor-product-db.yaml`
- Files: `k8s/prometheus/podmonitors/podmonitor-transaction-db.yaml`

**Example Manual PodMonitor for Transaction-DB:**
```yaml
apiVersion: monitoring.coreos.com/v1
kind: PodMonitor
metadata:
  name: transaction-db
  namespace: cart
  labels:
    app: transaction-db
    operator: cloudnativepg
spec:
  selector:
    matchLabels:
      cnpg.io/cluster: transaction-db  # Required: Must match cluster name
  podMetricsEndpoints:
  - port: metrics  # Port name (9187)
    interval: 15s
    scrapeTimeout: 10s
    path: /metrics
  podTargetLabels:
    - cnpg.io/cluster
    - cnpg.io/instanceRole
    - cnpg.io/instanceName
```

**Important Configuration Details:**
- `metadata.name`: Give your PodMonitor a unique name (typically matches cluster name)
- `spec.selector.matchLabels`: **Must use** `cnpg.io/cluster: <cluster-name>` label to correctly target PostgreSQL instances
- `spec.podMetricsEndpoints[].port`: Use `metrics` (port 9187)
- `spec.namespaceSelector`: Optional - use to specify namespace where PostgreSQL cluster is running (if PodMonitor is in different namespace)

**Benefits:**
- ✅ Full control over scrape configuration (intervals, timeouts, relabeling)
- ✅ Works with any CloudNativePG version
- ✅ Can be version-controlled separately
- ✅ No deprecation concerns
- ✅ Production-ready approach (official recommendation)
- ✅ Complete ownership of monitoring configuration

**TLS Support:**
